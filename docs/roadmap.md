# 职达迭代路线图

本文档记录当前架构审计的落地结果和后续板块边界。它不是对交付日期的承诺；新功能进入开发前，仍需明确用户问题、数据所有权、隐私范围和可验收指标。

## 已完成基线

- 职业域已从临时聊天内容拆分为简历、岗位、投递、面试、报告和技能六类结构化资源。
- 六个前端深链路由同一应用壳、导航和 API client 承载，具备 CRUD、加载、空状态、失败重试和移动端布局。
- 数据库迁移账本、用户所有权、CSRF、复合外键、主简历唯一性和用户级写入锁已建立。
- URL 抓取、文档导入、SSE 输出、日志与数据清理边界已经过安全收紧。
- 架构、用户、管理员、API、配置、部署、测试、安全与隐私文档已与代码对齐。

## 上线前技术门禁

| 项目 | 原因 | 完成标准 |
| --- | --- | --- |
| MySQL–Chroma 删除任务 | 用户锁可防并发，但两个存储仍无分布式事务 | 持久化删除任务状态、幂等重试、双侧校验与运维告警 |
| 大数据导出 | 当前全量 JSON 在内存中组装并持有用户锁 | 改为流式/异步导出，有配额、进度、过期和下载授权 |
| 真实数据库集成测试 | 单元测试不能完全替代 MySQL 约束和并发行为 | CI 覆盖两用户隔离、外键、唯一索引、回滚、重试与迁移恢复 |
| 资源配额 | 认证用户仍可创建大量记录、对话和向量 | 按用户/组织定义记录、存储、模型 token 和并发配额，超限可解释、可观测 |
| 依赖与可观测性 | 当前后端依赖版本较宽，运行指标有限 | 生成可重现锁定文件/SBOM，补充错误率、延迟、队列、配额和模型成本指标 |

## 建议新板块

### P1：求职执行闭环

1. **公司与联系人**：公司研究、招聘人/内推人、沟通记录和下次跟进。应关联岗位与投递，不把联系人信息写入自由聊天。
2. **日程与提醒**：投递截止、面试、跟进和 offer 截止日。先建站内任务/提醒模型，再接入外部日历，所有外发操作需明确确认。
3. **作品集与能力证据**：项目、成果、链接、量化指标和可复用 STAR 素材，为简历改写和面试回答提供结构化来源。
4. **Offer 与决策中心**：薪酬结构、地点、成长、风险、截止日和多方案对比。必须明示数据来源、假设和不确定性，不自动代替用户决策。

### P2：结构化 AI 与洞察

1. **可版本化的 AI 任务**：简历分析、岗位匹配、面试出题/评价直接产生有 schema 版本的报告，保存模型、提示版本、来源和生成时间。
2. **异步任务中心**：长文档解析、批量岗位导入、多份简历对比和大报告改为可取消、可重试、可查询进度的任务。
3. **漏斗与复盘洞察**：基于投递阶段事件计算转化率、停留时间和下一步建议。先定义可审计事件和指标口径，再做图表。
4. **岗位变化跟踪**：保存用户授权抓取的版本、差异和来源时间，不将当前网页当作永久可用的事实。

## 新板块准入规则

每个板块进入开发前应同时具备：

- 清晰的用户任务、页面入口、空/错误/移动端状态和删除语义。
- 带 `user_id` 的服务端所有权校验，以及必要的唯一、外键或事件约束。
- 追加式 migration、API 契约、配额/限流、审计与隐私说明。
- 单元、跨用户授权、真实 MySQL 约束、前端交互与容器冒烟测试。
- 可观测的成功指标和失败降级路径，不仅是“页面已存在”。
