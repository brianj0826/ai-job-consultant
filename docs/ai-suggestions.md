# AI 职业数据建议

AI 职业数据建议把对话中的可执行内容整理成结构化草稿。AI 只负责提出建议；用户检查并确认后，后端才会把数据写入简历、岗位、投递、面试、报告、技能或面试题业务表。

## 用户流程

1. 用户与职达 AI 对话。
2. 正常回答完成后，系统独立识别明确保存意图和高置信度行动建议。
3. 每条回答最多展示三张建议卡；低置信度、假设、示例和第三方经历不展示。
4. 用户可以编辑并保存待确认草稿、确认添加、忽略或恢复已忽略建议。
5. 确认时后端重新验证登录、CSRF、所有权、关系、枚举、唯一性和字段边界。
6. 创建成功后，卡片保留已添加状态和目标板块入口。

建议元数据会在确认前持久化，以支持刷新、切换会话和幂等重放；目标职业业务表只会在用户确认后写入。

## 支持范围

第一版只建议新增，不建议修改或删除已有数据。

| 类型 | 生成与确认边界 |
| --- | --- |
| 简历 | 只有存在完整正文时生成；默认不是主简历 |
| 岗位 | 需要明确岗位和足够 JD；默认状态为已收藏 |
| 投递 | 需要明确记录/投递语义；必须确认本人岗位 |
| 面试 | 需要明确安排或模拟语义；默认待开始，不生成评分 |
| 报告 | 支持简历分析、岗位匹配、面试复盘和职业计划 |
| 技能 | 默认待开始、进度 0；日期只来自用户明确表达 |
| 面试题批次 | 只能添加到已有面试；最多十题，批次原子写入 |

投递和面试题只在关系唯一明确时预选目标；有多个候选时，卡片保持待选择，用户必须在编辑器中选择本人记录后才能确认。当前账号没有任何可用关联记录时，系统不生成无法完成的建议。

面试题中的 AI 参考内容使用 `reference_answer` 和 `coaching_notes`；`answer`、`feedback` 和 `score` 继续表示用户真实练习结果。

第一版不支持跨建议依赖，例如不会在同一次确认中先创建岗位再创建投递。

## 状态与并发

建议状态为：

- `pending`：待确认，可编辑；
- `dismissed`：已忽略，可以恢复；
- `accepted`：已经创建目标记录，不可恢复或再次忽略。

每次编辑和状态变化都会增加 `revision`。客户端提交旧 revision 时返回 `409`，应刷新权威建议后重试。重复接受同一建议会返回第一次创建的结果，不会创建重复记录。

## AI 与故障边界

- 建议识别使用独立、无副作用的结构化模型调用。
- 模型输出不是可信输入；未知字段、伪造用户 ID 和未授权关系都会被服务端拒绝。
- 识别超时、无工具调用、坏 JSON 或校验失败时返回空建议，正常聊天不受影响。
- 未通过输出审核的 AI 回复不会产生建议。
- `CAREER_SUGGESTIONS_ENABLED=false` 可停止生成新建议；已有建议仍可查看和处理。

## 数据与隐私

建议可能包含简历、JD、报告和学习计划等敏感内容。它们存储在 MySQL 中，并：

- 绑定当前用户、会话和 AI 消息；
- 随会话或账号删除；
- 纳入职业数据 JSON 导出；
- 由“清空职业数据与知识库”一并删除；
- 不把 payload 写入应用日志。

为匹配关系，建议提取器只向模型发送当前用户必要的精简职业目录。部署方应在隐私告知中说明模型供应商处理对话与精简职业上下文的边界。
