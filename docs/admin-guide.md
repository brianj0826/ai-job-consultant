# 管理员指南

## 角色边界

| 能力 | user | admin | super_admin |
| --- | :---: | :---: | :---: |
| 使用个人工作台和职业数据 | 是 | 是 | 是 |
| 查看管理概览、用户、反馈和审计 | 否 | 是 | 是 |
| 启用/停用普通用户 | 否 | 是 | 是 |
| 修改管理员或超级管理员状态 | 否 | 否 | 是 |
| 分配角色 | 否 | 否 | 是 |
| 重置密码 | 否 | 否 | 是 |

管理 API 默认不提供浏览用户职业数据或知识库正文的能力。数据库、备份和主机管理员仍可能在系统外接触数据，应通过最小权限和运维制度约束。

## 初始化第一个超级管理员

首次启动前，在安全的部署环境中同时设置：

```dotenv
ADMIN_BOOTSTRAP_USERNAME=<initial-admin>
ADMIN_BOOTSTRAP_PASSWORD=<strong-one-time-password>
```

后端只在不存在超级管理员时执行引导。确认账号可登录后，立即从部署环境移除两个明文变量，并按组织策略更换密码。不要把这些值提交到仓库、镜像或工单。

如果两个变量只设置了一个，启动配置应视为错误；已有超级管理员时，保留变量也不会自动提升其他账号，但仍构成不必要的密钥暴露。

## 控制台

规范页面为：

- `/admin/overview`：用户、会话、消息与反馈概览。
- `/admin/users`：搜索、分页、启停账号、角色与密码管理。
- `/admin/feedback`：查看被点赞/点踩的 AI 回复片段。
- `/admin/audit-logs`：按操作类型查看管理行为。

旧 `/admin` 与 `/admin/audit` 路径只作为前端重定向兼容入口。

## 账号操作

### 启用或停用

停用账号会撤销该用户现有认证会话。系统隐藏“账号不存在”和“账号已停用”的登录差异，避免枚举账号状态。管理员不能修改自己的状态；特权账号和最后一个启用的超级管理员受到额外保护。

停用不等于删除：用户的聊天、职业数据和知识库仍保留。需要数据处置时，应使用经授权的单独流程，不能把停用当作隐私删除。

### 分配角色

只有超级管理员可以修改角色。变更前应确认业务需要，并定期复查特权账号。不要把共享账号设为管理员。

### 重置密码

超级管理员触发重置后，后端生成一次性展示的临时密码、撤销目标用户会话，并设置强制改密。临时密码只在当前响应出现一次：

1. 通过独立的安全渠道交付给用户。
2. 不在聊天、工单或审计备注中长期保存。
3. 要求用户首次登录立即改密。

管理员无法查看用户原密码。

## 审计与日常检查

管理写操作和管理员登录事件进入审计日志。建议至少检查：

- 新增或变更的特权账号。
- 大量账号停用、密码重置或异常登录。
- 审计记录中的来源 IP 与 User-Agent 是否符合预期。
- `/api/health/ready`、容器重启次数、应用错误日志和磁盘空间。
- MySQL 与 ChromaDB 最近一次同批备份是否可恢复。

审计日志不是防篡改的外部合规日志；需要更强保证时，应把结构化日志发送到受控、只追加的集中平台。

## 管理员安全要求

- 生产环境只通过 HTTPS 访问，不公开后端、MySQL、Redis 或 Adminer。
- 使用独立个人账号；超级管理员只用于必须的操作。
- 完成 Adminer 临时维护后关闭 `tools` profile。
- 不把用户内容复制到未经批准的第三方服务。
- 角色、密码重置、备份恢复与批量数据操作应遵循部署方审批流程。

完整的环境和恢复操作见[配置参考](configuration.md)与[部署指南](deployment.md)。
