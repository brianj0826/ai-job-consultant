# 安全说明

## 支持范围

安全修复以当前维护的 `main` 和明确标记的发布版本为目标。旧提交、个人分支、修改后的镜像和未经维护者确认的第三方部署不保证获得补丁。部署方应固定提交或镜像摘要，并持续跟踪更新。

## 报告漏洞

请不要在公开 Issue、讨论区、聊天截图或日志中披露可利用细节、账号数据、Cookie、API 密钥或简历内容。

优先使用仓库的 GitHub Private Vulnerability Reporting / Security Advisory 私密渠道；如果该功能不可用，请通过仓库所有者已经公开的私密联系方式请求安全报告渠道。报告建议包含：

- 受影响提交/版本和部署形态。
- 最小复现步骤、预期与实际结果。
- 影响范围与是否需要已登录、管理员或网络位置。
- 已做的数据访问或变更；只使用测试账号和最少数据。
- 建议修复或缓解方式（如有）。

维护者确认接收前不要扩大测试范围。没有明确授权时，不得扫描生产实例、访问其他用户数据、执行拒绝服务、持久化、社会工程或破坏性验证。

## 当前安全边界

项目内置以下控制，但不构成绝对安全保证：

- 服务端随机 Cookie 会话；认证 Cookie 为 HttpOnly，数据库只保存 token 摘要。
- 双提交 CSRF；所有受认证的状态变更接口要求 `X-CSRF-Token`。
- 服务端资源归属校验和 `user` / `admin` / `super_admin` RBAC。
- Redis 共享的登录/业务限流；生产拒绝进程内 fallback。
- PBKDF2-SHA256 带盐密码哈希和旧哈希登录后升级。
- 管理操作审计、账号停用/改密后的会话撤销与最后超级管理员保护。
- URL 抓取的 scheme、DNS/IP、重定向和响应大小限制，用于降低 SSRF 与资源耗尽风险。
- 上传扩展名、字节、PDF 页数、提取字符和分块上限。
- 报告 payload 的紧凑 JSON UTF-8 编码限制为 256 KiB，降低单条记录的存储与解析滥用风险。
- 职业写入、职业导出与知识库写入使用 MySQL 用户级 advisory lock，与用户数据清空串行；主简历唯一和岗位归属另由数据库唯一/复合外键约束兜底。
- 非流式与 SSE 回复使用同一输出审核；SSE 会先缓冲完整模型回复，审核后才释放 token。
- Cookie/CORS/可信代理生产配置校验。

安全模型假设主机、容器运行时、MySQL、Redis、Chroma 卷、反向代理和 secret store 由可信运维人员控制。拥有这些系统管理权限的人可能绕过应用层授权。

## 部署方必须完成

- 只通过 HTTPS 暴露前端入口；不要公开后端、MySQL、Redis 或 Adminer。
- 使用独立非 root 数据库账号、强随机密钥、最小网络出口和受限代理 CIDR。
- 把 `.env`、DeepSeek 密钥、数据库密码和引导密码放入受控 secret store，并建立轮换流程。
- 在基础设施层阻断 localhost、私网、link-local、组播、保留地址和云 metadata 的非预期出口；应用 SSRF 检查不是网络 ACL 的替代品。
- 对上传和文档解析使用容器隔离、资源限制、依赖更新和恶意文件处置流程。扩展名与大小检查不能证明文件无害。
- 集中收集最少必要日志，屏蔽认证材料和用户正文，设置访问、保留与告警策略。
- 加密并定期恢复演练 MySQL + Chroma 同批备份。
- 在模型供应商配置中核对数据使用、保留、地域和访问条款。

生产配置示例和边界见[配置参考](docs/configuration.md)与[部署指南](docs/deployment.md)。

## 已知限制

- 遗留 bootstrap 与自有版本 migration 共用 schema lock，迁移账本记录 SHA-256 checksum，但尚未采用成熟的通用 migration 工具链。
- 用户级 advisory lock 只能协调遵守该协议的应用请求，不能形成 MySQL—Chroma 分布式事务；清空失败需重试并核验两侧。
- 职业导出会在持锁期间把该用户全部结构化数据组装到进程内存，不适合无上限的大数据账号。
- readiness 当前检查 MySQL 与限流后端，不验证每次 DeepSeek、Chroma 或网页访问。
- 应用内内容审核规则有限，不能替代专业安全、合规或人工审核。
- AI 输出可能包含不准确或不当内容；引用来源不代表结论已验证。
- 管理审计保存在同一 MySQL 中，不是外部防篡改账本。
- RAG 文本片段会存入 Chroma，相关片段可能随提示词发送给模型供应商。

发现新限制时，应在修复、测试或缓解措施可用前明确记录，而不是用文档暗示不存在风险。

## 依赖与变更

- 提交前运行[测试指南](docs/testing.md)中的对应门禁。
- 依赖升级应检查上游安全公告、锁文件差异、镜像来源与回归测试。
- 安全相关配置变更应经过双人审查；不要在日志、PR 描述或 CI artifact 中粘贴真实秘密。
- 紧急缓解也应保留可审计的提交，修复后补充回归测试与影响说明。
